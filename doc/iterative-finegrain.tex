\subsection{Implementation}
\label{sec:iter-fine-impl}

The \textbf{fine grain iterative} strategy extends the Tile version by exploiting the
additional parallelism that exists \emph{inside} each tile.
Rather than completing the border check and fill/compute sequentially within each tile
task, the fine grain version exposes three independent sub-tasks per tile:

\begin{enumerate}
    \item A task that computes the \emph{vertical} borders of the tile (left and right
          columns), setting \texttt{eV}.
    \item A task that computes the \emph{horizontal} borders (top and bottom rows,
          including all four corners), setting \texttt{eH}.
    \item A task (spawned after \texttt{\#pragma~omp~taskgroup} ensures both border tasks
          have finished) that either fills the tile uniformly or, in the non-uniform case,
          spawns one child task per row to compute all interior pixels.
\end{enumerate}

The outer tile task from the Tile version is \textbf{kept} as a wrapper; the three inner
tasks are created inside it.

The key parallel section (three inner tasks per tile: vertical borders, horizontal borders, and fill/compute) is implemented in \texttt{codis/fine-grain.c}.

The \texttt{\#pragma~omp~taskgroup} enforces that the result tasks begin only after
\emph{both} border tasks complete, because the fill/compute decision depends on
\texttt{eV} and \texttt{eH}.
Histogram increments remain protected by \texttt{\#pragma omp atomic}.
Display updates use \texttt{\#pragma omp critical} to serialize X11 calls.

\paragraph{Comparison with Iterative Tile.}
The Tile version creates $\bigl(\mathit{ROWS}/\mathit{TILE}\bigr)^{2}$ tasks in total.
The Fine Grain version creates up to $3 + 2\cdot\mathit{TILE}$ tasks \emph{per tile},
increasing the task count by roughly $2\cdot\mathit{TILE}$ fold.
This exposes more parallelism at the cost of higher task-creation and synchronisation
overhead; whether the trade-off is beneficial depends on tile size and thread count.

\paragraph{Observations.}
The fine grain version scales remarkably well across all thread counts.
At 32 threads the elapsed time drops from 3.49\,s to 0.13\,s, yielding a
speedup of $26.28\times$ and a global efficiency of 82\,\% (Table~\ref{tab:fg-exec}).

The efficiency breakdown (Table~\ref{tab:fg-efficiency}) shows that the dominant loss
source is \emph{load imbalance}: the load-balancing factor falls to 87.67\,\% at 32
threads while in-execution efficiency remains above 95\,\%.
IPC and instruction scalability are virtually perfect ($\geq$99.8\,\%), confirming that
the computation itself is highly regular and portable across cores.

The task statistics (Table~\ref{tab:fg-tasks}) reveal 8\,448 explicit tasks in total,
providing ample work to keep all threads busy.
Synchronisation overhead grows from 0\,\% at 1 thread to 9.28\,\% at 32 threads,
while scheduling overhead reaches 7.59\,\%.
These costs are still modest relative to the useful computation per task
(\(\approx\)412\,\textmu s average), so the task granularity remains appropriate even at
high parallelism.
The slight decline in load balance at 32 threads arises because the 8\,448 tasks do not
divide evenly among threads and some tiles are computationally heavier (Mandelbrot
border regions).

Compared with the Iterative Tile version, the fine grain decomposition exposes
substantially more parallelism by splitting each tile's border and interior work into
independent tasks.
This allows threads to overlap border computation across different tiles and fills
in potential idle gaps, leading to the near-linear speedup curve observed here.
